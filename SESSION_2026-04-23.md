## Work Completed This Session

### 1. Clarified ref_reference vs ref_item Distinction

**Key insight delivered**: These are fundamentally different abstractions:

**ref_item** = Physical products in inventory
- Has `partnum` (Epicor ERP integration)
- Base pricing and descriptions
- Represents "what exists"

**ref_reference** = Selectable options in configuration contexts  
- References ref_item (BUT `ref_item_UID` can be NULL)
- Has `ref_categories_UID` for namespace/context
- Inventor CAD mappings (`inv_key`, `inv_value`)
- Represents "what can be selected where"

**Value of abstraction**:
1. Same item in multiple contexts with different pricing/mappings
2. Virtual options (NULL ref_item_UID) - "None", "N/A", calculated values
3. Context-specific overrides without duplicating items
4. Validation rules target **choices** (ref_reference) not products (ref_item)
5. Integration flexibility - same item maps differently to Inventor per context

**Bottom line**: ref_item = catalog of things; ref_reference = catalog of choices.
