# Development Session - April 25, 2026

## Project: Maximum Panel Length Application

### Session Summary
Initial planning and architecture design for a web application to determine maximum architectural panel length for Norex panels.

---

## Project Overview

**Purpose:** Quick reference tool for finding maximum panel length based on product specifications.

**Users:** Internal team (St-Hyacinthe and Strathroy plants)

**Input Parameters:**
- Plant Location: STH (St-Hyacinthe) or SRY (Strathroy)
- Panel Color
- Paint Type: PVDF CLASSIC, PVDF SIGNATURE, SMP CLASSIC, SMP PREMIUM, METALLIC
- Gage: 22, 24, 26
- Profile: Grooved, Silkline, Microribbed, None
- Finish: Embossed, Smooth

**Data Sources:**
- MaxLength-STH.csv
- MaxLength-SRY.csv

---

## Architecture Decisions

### Technology Stack
- **Backend:** Python + FastAPI + Uvicorn + Pydantic + pandas
- **Frontend:** React + Vite + Tailwind CSS + Axios
- **Deployment:** Docker + Docker Compose
- **Testing:** pytest (backend), Playwright/Cypress (E2E)

### API Design
**Endpoints:**
- `GET /api/plants` - List plants (STH, SRY)
- `GET /api/plants/{plantId}/data` - Get complete dataset for plant
- `GET /docs` - Swagger UI
- `GET /redoc` - ReDoc

**Key Decision:** Full dataset to frontend (Option A)
- Dataset is small (~50-100 rows, <50KB)
- Single API call provides all data for session
- Enables smart UI filtering (show only valid combinations)
- Frontend calculates results instantly
- Simpler backend (pure data serving, no validation logic)

### Data Strategy
- **Static data:** CSV files represent material physics, rarely change
- **Loading:** Parse CSVs at startup with pandas, hold in memory
- **Full dataset transfer:** Backend serves complete plant data to frontend
- **Client-side processing:** Frontend filters options and calculates results from local dataset
- **Updates:** Replace CSV files and redeploy containers (no hot reload)

### MVP Principles
1. No internationalization (internal tool, display bilingual labels from CSV as-is)
2. NA is valid result (not an error, display clearly to user)
3. Docker Compose for both dev and prod (VM in VPN)
4. Backend unit tests + E2E tests for critical flows
5. No logging infrastructure for MVP
6. Simple deployment: replace CSV → rebuild → restart

## UI/UX Design Decisions

### Style Reference
- **Inspiration:** ToitCalc Pro (see References/UI style example.png)
- **Aesthetic:** "Toy but pro" - playful yet professional
- **Layout:** Left panel for parameters, right panel for results
- **Design:** Dark navy header, blue accent buttons, card-based layout, rounded corners

### Interaction Model

**Selection Flow:**
1. **Plant:** Toggle switch (STH ↔ SRY)
2. **Color:** Scrollable list (PRIMARY - selected first)
3. **Other characteristics:** One-click button groups (NOT dropdowns)
   - Paint Type: Button group
   - Gage: Button group (22/24/26)
   - Profile: Button group (Grooved/Silkline/Microribbed/None)
   - Finish: Button group (Embossed/Smooth)

**Dynamic Filtering Logic:**
- User selects Color first from scrollable list
- Paint/Gage/Profile/Finish buttons dynamically enable/disable based on available options for selected color
- When characteristic has zero valid options, show "no options available" visual state
- Each selection cascades to update available options for remaining characteristics

**Result Display:**
- Large prominent number with units (feet)
- Show selected parameters for context
- NA values displayed clearly as "Not Available"

### UI Interaction Model (Detailed)

See [ui-specifications.md](ui-specifications.md) for complete specifications.

**Control Types:**
- **Plant:** Toggle switch (STH ↔ SRY)
- **Color:** Scrollable list (primary selection)
- **Paint/Gage/Profile/Finish:** One-click button groups

**Three-State System:**
- **Selected:** Active/highlighted
- **Unselected (third state):** No selection made
- **No options available:** Disabled/grayed out

**Parameter Hierarchy:**
```
Plant (Level 0)
  ↓
Color (Level 1) - primary filter
  ↓
Paint, Gage, Profile, Finish (Level 2)
```

**Smart State Preservation:**
- **Plant change:** If selected color exists in new plant → keep it, else reset to third state
- **Color change:** If selected characteristics exist for new color → keep them, else reset to third state
- **Clear button:** Reset ALL parameters (including plant) to third state

## Key Decisions Log

| Decision | Rationale |
|----------|-----------|
| Full dataset to frontend | Small data (<50KB), enables smart filtering, instant results |
| Client-side filtering | Show only valid combinations based on actual data |
| No calculate endpoint | Frontend has all data, can calculate locally |
| Simple backend | Pure data serving, no business logic |
| Single Docker host | Both containers on same instance with private network |
| Backend not exposed | Only accessible via Nginx proxy, secure internal communication |
| No i18n | Internal tool, bilingual labels already in CSV |
| NA as data value | Part of dataset, not an error condition |
| Docker Compose | Consistent dev/prod environment |
| Python + FastAPI | Fast development, excellent API docs |
| React + Tailwind | Modern stack, rapid UI development |
| One-click controls | Better UX than dropdowns for limited options |
| Color-first selection | Primary filter that determines other options |
| Three-state system | Unselected, selected, no options available |
| Smart state preservation | Keep selections when valid during plant/color changes |
| Clear button resets all | Including plant selection, complete reset |

## Questions & Answers

**Q:** Should we use server-side or client-side filtering?  
**A:** Client-side. Data is small/static, provides faster UX.

**Q:** Handle NA values?  
**A:** Display as valid result "Not Available" - user needs this information.

**Q:** Deployment strategy?  
**A:** Docker Compose for both dev (localhost) and prod (VM in VPN).

**Q:** Data update mechanism?  
**A:** Replace CSV files → rebuild containers → restart. No hot reload needed for MVP.

**Q:** Folder naming?  
**A:** `src/` (lowercase), `References/` (for documentation, not application resources).
